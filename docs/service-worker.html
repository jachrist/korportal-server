Service Worker og PWA

Caching, offline-støtte, oppdateringsflyt og manifest

Oversikt

sw.js gir offline-støtte og raskere lasting. sw-register.js registrerer den og håndterer automatisk oppdatering. Brukeren trenger ikke gjøre noe – nye versjoner lastes automatisk.

Cachestrategi

Service Workeren bruker to strategier:

  • Cache first: For statiske filer (CSS, JS, bilder, fonter). Serverer fra cache umiddelbart, oppdaterer i bakgrunnen ved neste deploy.
  • Network first: For HTML-sider og dynamisk innhold. Prøver nettverk først for å sikre fersk data, faller tilbake til cache ved offline.
Merk: env.js er eksplisitt ekskludert fra cache first. Funksjonen isStaticAsset() i sw.js sjekker dette.

Cachede filer

Tre cache-botter brukes:

  • CACHE_NAME (master-cache): Brukes for versjonsstyring.
  • STATIC_CACHE: Alle HTML, CSS, JS, manifest.
  • DYNAMIC_CACHE: HTML og data som caches ved første besøk.

Oppdateringsflyt

  1. Endre filer i kodebasen.
  2. Bump CACHE_NAME, STATIC_CACHE og/eller DYNAMIC_CACHE versjonsnummer i sw.js.
  3. Deploy til Azure Static Web Apps.
  4. sw-register.js oppdager ny SW, installerer den.
  5. Ny SW aktiveres og sletter gamle cacher.
  6. Siden reloades automatisk med nye filer.
Advarsel: Hvis du glemmer å bumpe cacheversjonen, kan brukere få gammel kode fra SW-cache selv etter deploy.

manifest.json

PWA-manifestet gjør appen installerbar. Viktige egenskaper:

  • name – Fullt appnavn
  • short_name – Kort navn for hjemskjerm
  • start_url – /
  • display – standalone
  • theme_color – #0b1220

Ikoner fra 72px til 512px er inkludert for ulike enheter og kontekster.

Feilsøking

  • Brukere får gammel versjon: Bump CACHE_NAME i sw.js og deploy på nytt.
  • SW registreres ikke: Sjekk at sw.js ligger i roten (ikke i js/). SW-scope er begrenset til sin egen katalog og under.
  • Cache tømmer seg ikke: I DevTools > Application > Storage > Clear site data. Eller send 'clearCache'-melding til SW via konsollen.
